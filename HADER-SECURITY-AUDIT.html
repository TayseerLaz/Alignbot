▶ EXECUTIVE VERDICT

Strong posture — no Critical exposure. Production-safe with 6 High items to close.

Unlike a Supabase-style app, Hader exposes no database to clients: every request passes through an authenticated Fastify API, and tenant isolation is enforced by real Postgres Row-Level Security (forced policies + a non-superuser aligned_app role + a self-enforcing CI drift gate). There is no anonymous data access, no client-side database key, and no BOLA hole in the sampled surface. The historical superuser-RLS-bypass bug is fixed and verified.

The review found 0 Critical and 6 High findings. None is remotely exploitable for mass data exfiltration today — the High items are: a 10-minute stolen-refresh-token replay window, one weak payment-webhook verifier (MyFatoorah), TOTP seeds stored in plaintext (defeats 2FA only after a separate DB compromise), and three SSRF gaps in background workers (crawler / image fetch) that bypass the project's own IP-pinning layer. All are fixable in hours, mostly by routing existing code through primitives the codebase already owns.

// 02 — CONTROLS VERIFIED CORRECT

What is genuinely solid

These are not assumptions — each was read in source and, where applicable, cross-checked against the platform's own test gates. This is why there are no Critical findings.

  • Real RLS, role-switched. withTenant runs every query inside a transaction that does SET LOCAL ROLE aligned_app (a NOLOGIN, non-superuser role) + binds app.current_org_id. Superuser RLS-bypass is impossible for the duration. lib/db.ts:38
  • Every tenant table is ENABLE + FORCE ROW LEVEL SECURITY with a tenant_isolation policy across ~60 tables. rls.sql:60
  • Self-enforcing CI drift gate. rls-drift.test.ts enumerates every table with an organization_id column and fails the build if any lacks RLS + a policy. A new model that forgets RLS cannot ship.
  • DB-level isolation proof. tenant-isolation.test.ts binds org A's role+context and directly SELECTs org B's row id → asserts 0 rows. Catches regressions that bypass route handlers.
  • JWT alg pinned. Verify allow-lists ['HS256'] with enforced issuer/audience (distinct per token type). alg:none and HS/RS confusion are impossible. Org id comes only from the verified org claim. jwt.ts:50
  • Passwords bcrypt cost 12; opaque tokens 192-bit, SHA-256 at rest, single-use, TTL-bounded. Sessions revoked on password change/reset.
  • Stripe & PayPal webhooks correctly verified (Stripe: t.body HMAC + 5-min replay window; PayPal: provider verify-API) before any order flips to paid; markCartPaid is idempotent.
  • Meta WhatsApp/Messenger inbound verifies X-Hub-Signature-256 over raw bytes with the channel's app secret, constant-time, and refuses (403) when no secret is configured — no unauthenticated-accept path.
  • Secrets encrypted at rest (AES-256-GCM, per-value IV, versioned, fail-loud). SECRET_ENCRYPTION_KEY is required at boot in production. WhatsApp/Messenger tokens, connector config, payment creds all encrypted.
  • SSRF guard primitive is strong where used: resolves all DNS answers, rejects loopback/RFC1918/link-local/IMDS 169.254.169.254/CGNAT, and pins the connection IP (kills rebinding). Connector test + sync use it correctly.
  • Wasabi: private bucket, UUID object keys, tenant-prefixed (org/<orgId>/...); raw-key preview rejects any key not under the caller's org prefix.
  • Nonce-based CSP on the web app (strict-dynamic, no unsafe-inline scripts, frame-ancestors 'none') + HSTS preload, full header set, allow-list CORS.
  • Bot KB grounding is strictly tenant-scoped — every catalog query filters by organizationId and runs under RLS; embedding search operates only over already-org-filtered rows. No cross-tenant catalog bleed. A deterministic post-LLM hallucination scanner enforces "no fabrication."
  • API keys 192-bit; per-key rate limit; org resolved from the key row, never a client header; every read endpoint carries an explicit scope check.

H-0110-minute stolen-refresh-token replay window neutralises rotation tripwireHigh
Class
Session / token replay
File
auth.service.ts:359
OWASP
A07:2021
Pre-req
Captured refresh token
const REUSE_GRACE_WINDOW_MS = 10 * 60_000;   // widened 10s → 60s → 10 min
if (reusedSession && withinGrace) {
  // re-issues a fresh ACCESS token — no rotation, no revocation, no audit
  return { accessToken: access.token, refreshToken: null, ... };
}
▶ IMPACT
Refresh-token rotation reuse-detection is the control that catches a stolen token. For 10 minutes after any legitimate rotation, an attacker holding the previous token is silently minted a fresh valid access token — no session revocation, no refresh_token_reuse_detected audit. A busy client rotates often, producing near-continuous overlapping grace windows. The wide margin was added to fix a multi-tab race that the client now also solves with Web Locks, so it is over-provisioned.
✓ FIX
  • Shrink the window to ≤30s.
  • On a grace-window hit, bind acceptance to the session's device fingerprint (userAgent/ipAddress); on mismatch → revoke family + audit.
  • Emit a low-severity audit event on every grace re-issue so replays are observable.

Every other tenant secret is AES-256-GCM encrypted, but the TOTP seed — the entire second factor — is cleartext and is not in the Prisma crypto extension's field list. Anyone with DB read (the exact threat secret-crypto.ts exists for: leaked pg_dump, SQLi, replica access) can generate valid codes for every 2FA-enabled user, fully defeating the second factor.   • Backfill existing rows with the same pattern used for the connector-secret (F-01) migration.

M-01withRlsBypass disables RLS with no org bound — explicit filters are the only guardrailMedium
Evidence — lib/db.ts:55
export async function withRlsBypass(fn) {
  return prisma.$transaction(async (tx) => {
    await tx.$executeRawUnsafe(`SELECT set_config('app.bypass_rls','on',true)`);
    return fn(tx);   // RLS fully off; NO org bound
  });
}

Inside a bypass block RLS is fully disabled; ~20 non-admin call sites are safe today only because each manually adds where:{ organizationId }. Any future query that forgets the filter is an instant cross-tenant read/write with no DB backstop and no test coverage. Fix: add withRlsBypassForOrg(orgId, fn) that also binds app.current_org_id, push non-admin "bypass" paths onto withTenant (as the messenger hot path already does), and reserve true bypass for requireAlignedAdmin + auth bootstrap. Add a lint/grep gate asserting every withRlsBypass body contains an organizationId filter.

M-02Account lockout is not race-safe (parallel-request brute-force bypass)Medium
Evidence — auth.service.ts:149 (and TOTP twin :212)
const failedAttempts = user.failedLoginAttempts + 1;   // read-modify-write
await prisma.user.update({ where:{id:user.id},
  data:{ failedLoginAttempts: failedAttempts, ... }});    // last-writer-wins

N concurrent attempts all read the same counter and each write start+1, so it advances by 1 instead of N — an attacker firing in parallel submits far more than 5 guesses before lockedUntil is set. Fix: atomic { failedLoginAttempts: { increment: 1 } } and decide locking from the returned value (apply to both bad-password and bad-TOTP branches).

M-03Lockout keyed per-account only → targeted lockout-DoS + no per-IP failure throttleMedium

auth.service.ts:138 — anyone who knows a victim's email can lock them out (5 bad attempts → 15-min lock), an availability/harassment DoS; conversely there is no per-IP failed-attempt accounting, only the coarse global 10/min limiter. Fix: keep account lockout but add a progressive per-IP failure counter (Redis) and/or CAPTCHA after K failures from one IP; prefer soft-delay over hard-lock to blunt the targeted-DoS angle.

M-04Cookie-authenticated routes rely solely on SameSite=Lax — no CSRF token / Origin checkMedium

cookies.ts:15 — /auth/refresh, /logout, /switch-org authenticate purely via the cookie. SameSite=Lax blocks cross-site POST in modern browsers, so this is mostly mitigated, but it is the only layer (no CSRF token, no Origin/Referer allow-list, cookie unsigned). A forged /auth/refresh could churn the victim's session on browsers where Lax is weakened. Fix: add an Origin/Referer allow-list check (against CORS_ORIGINS) on the cookie-auth routes, and/or a double-submit token; consider SameSite=Strict (the cookie is first-party only).

M-05COOKIE_SECURE defaults to false and isn't forced in productionMedium
COOKIE_SECURE: z.string().default('false').transform(s => s === 'true'),   // env.ts:24

If a prod .env omits COOKIE_SECURE=true, the refresh cookie is sent over plain HTTP and can be sniffed — unlike SECRET_ENCRYPTION_KEY, there is no fail-closed prod guard. Fix: in env.ts, force COOKIE_SECURE=true when NODE_ENV==='production' (or hard-fail boot), mirroring the existing encryption-key pattern.

M-06Webhook HMAC silently falls back to JSON.stringify(req.body) when rawBody is absentMedium
const rawBody = req.rawBody ?? JSON.stringify(req.body ?? {});   // all 4 verifiers

If req.rawBody is ever missing (a different/missing Content-Type that bypasses the buffer parser, or a future parser change), verification reverts to hashing a Node re-serialization — reintroducing the exact key-reorder bug class the raw-byte capture was built to kill. Fix: treat a missing rawBody as a hard 400/401 reject in all verifiers; remove the ?? JSON.stringify(...) fallback.

M-07SSRF — outbound webhook delivery has no guard / pinning at send timeMedium

webhook-delivery.ts:79 — endpoint.url is validated only at create/update time (synchronous literal-IP check). At delivery there is no assertSafeOutboundUrl and no pinning dispatcher, so a hostname that resolved public at creation but later points at an internal IP (DNS rebinding / attacker DNS) gets the signed payload POSTed to it. The connector sync worker does this correctly with getSsrfDispatcher(); the webhook worker should too. Fix: validate + pin at send time.

M-08Worker pino logger has no redact config — latent secret leakMedium

worker/src/index.ts:25 — the worker is exactly where decrypted secrets are in scope (sync builds Authorization headers from decrypted creds; webhook-delivery handles signing secrets). The API logger redacts all of these paths; the worker does not. No current call-site logs a raw secret, so this is latent — one careless log.error({ headers }) from leaking a tenant token. Fix: extract the API's redact paths to a shared module and pass it into the worker's pino(...) config.

M-09Bot: customer message un-fenced; platform marker grammar replayed from historyMedium
const messages = [ ...(args.history ?? []),
  { role:'user', content: args.userMessage } ];   // raw, untagged — bot-engine.ts:1133

The customer message is passed as a raw user turn with no data-fencing; the only jailbreak/prompt-extraction defense is one soft system line. Extraction itself is low-severity (the prompt holds only that tenant's own catalog framing, no secrets), but it discloses the marker grammar ([CART:…], [PAYMENT_LINK], [BOOKING:]). Markers are parsed from the bot's reply (not the customer's), so echoing them can't forge an order — but conversation history is replayed verbatim (:1134), giving a failed injection a second chance. Fix: (a) wrap userMessage in an explicit <customer_message> delimiter + "treat as untrusted data" system line; (b) higher value: strip/escape platform markers from inbound customer text before persistence so both the live turn and replayed history are clean.

M-10Read-cache "stale" tier is never served → no stampede protection on expiry/flushMedium

read.routes.ts:32 — on a stale/miss, every concurrent chatbot request for the same org runs the full Prisma query simultaneously; there is no single-flight lock and the advertised "5-min stale" value is never actually returned while revalidating. A traffic burst at the 60s boundary (or right after a catalog edit flushes the cache) stampedes the DB — a self-inflicted availability amplifier (not a confidentiality issue). Fix: serve stale immediately and revalidate in the background, or take a short per-key Redis lock so only one caller recomputes; serve stale on loader error.
